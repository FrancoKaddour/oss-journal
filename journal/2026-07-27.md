## Zag #3203 — dialog backdrop click-through (debut)

Root-caused a modal-dialog bug: `disablePointerEventsOutside` sets `body { pointer-events: none }` and relies on inheritance, but a background descendant with an explicit `pointer-events: auto` overrides the inherited `none` and stays clickable under the backdrop (CSS spec behavior). Confirmed the exact line and the layer-stack architecture.

The robust fix is architectural — migrate to the native `inert` attribute on the top-blocking layer's siblings (Radix's approach), which CSS can't override. Since that's a design choice the maintainer should weigh in on, entered the repo with an analysis + proposed direction + offer to implement with an e2e test, rather than a cold architectural PR. First contribution to the Ark/Zag ecosystem. [Comment](https://github.com/chakra-ui/zag/issues/3203#issuecomment-5070431222).

Note: fourth bug this week (with #11038, #35546) where the root cause is clear but the fix is architectural — a signal that in well-maintained repos, the easy bugs are already taken.

## Moonshot #35546 CONVERGED — connected to #35528 with empirical proof

Retook the floating-ui/Ark UI positioning regression, this time with a real repro (Storybook 10.5.3 + Ark UI React, built via `storybook build`, inspected with Playwright). Key discovery: it reproduces only for focus-managing components. A `Menu` parks off-screen; a `Popover` (no focus move) positions fine — which is why the reporter's focus-patch theory looked ruled out.

Captured stack proves the mechanism:
```
TypeError: Illegal invocation
    at HTMLElement.get [as focus]   // Storybook 10.5 focus accessor
    at trackFocusVisible            // @zag-js/focus-visible
```
Zag's `trackFocusVisible` reads `HTMLElement.prototype.focus`; the 10.5 getter brand-checks on the prototype receiver and throws; the uncaught error aborts Zag before floating-ui positions → positioner stuck at `-100vh`.

Proved the fix empirically: patched the built bundle with #35528's guard (return the installed method for prototype reads). Same Menu story went from `matrix(1,0,0,1,0,-720)` + `pointer-events: none` + thrown error → `matrix(1,0,0,1,16,45)` + `pointer-events: auto` + no error. So #35528 resolves three issues: #35503, #35502, #35546. Posted the diagnosis on both the issue and the PR.

## Storybook — focus instrumentation fix (issue #35503)

Root-caused an uncaught `TypeError: Illegal invocation` on first Docs-page load. The issue's own theory pointed at react-aria and cross-realm iframes, but the stack trace pointed elsewhere: `HTMLElement.get [as focus]` — a getter installed by Storybook's own focus instrumentation (added in 10.5).

The instrumentation defines `focus` as an accessor on `HTMLElement.prototype`, and its getter starts by touching `this.ownerDocument`. Focus-management libraries (react-aria, Zag) capture the method by reading it off the prototype before wrapping it — with the prototype as receiver, the native `ownerDocument` accessor brand-checks and throws.

- Fix: the getter now hands back the currently installed method for prototype reads.
- Tests reproduce the full capture-and-wrap pattern (red-green verified against the old implementation).
- Likely also explains #35502 (Zag focus-visible breaking under the same instrumentation).
- PR: [storybookjs/storybook#35528](https://github.com/storybookjs/storybook/pull/35528)

Also marked [#35527](https://github.com/storybookjs/storybook/pull/35527) (JSDoc code fences) ready for review.

## TanStack Router — issue #7829 triage (`wrapInSuspense` ignored in solid-router)

Confirmed the divergence: solid-router hardcodes `ResolvedSuspenseBoundary = () => Solid.Suspense` and never reads `route.options.wrapInSuspense`, while react-router resolves the boundary conditionally. Validated the reporter's proposed fix locally with one parity addition (react's `errorComponent?.preload` arm), ran the solid-router unit suite (162 tests green + clean typecheck), and noted the coverage gap: no dedicated `wrapInSuspense` tests exist in either adapter.

Posted the validation on the issue: [TanStack/router#7829](https://github.com/TanStack/router/issues/7829#issuecomment-5012077371). Fix branch ready locally for when it fits the team's release schedule.
